## 3. count=0 영역에서 색상이 보이는 이유

### Observation
- count=0인 텍셀에서도 최종 텍스처에서 색상이 보임

### Code-level behavior
```python
filled = weight_np > 0

baked = np.where(
    filled[:, :, np.newaxis],
    accum_np / np.maximum(weight_np[:, :, np.newaxis], 1.0),
    0.0,
)
```

- count=0 텍셀은 초기값 0.0 (검정)

이후 nearest-neighbor 보정 수행:

```python
_, nearest = distance_transform_edt(~filled, return_indices=True)

rows_nn = nearest[0][~filled]
cols_nn = nearest[1][~filled]

baked[~filled] = baked[rows_nn, cols_nn]
```

또한 seam 인근 영역에 대해 추가 보정:

```python
dilated = binary_dilation(filled, iterations=SEAM_DILATION)
seam_mask = dilated & ~filled

result[seam_mask] = baked[nearest[0][seam_mask], nearest[1][seam_mask]]
```

### Interpretation
- count=0 텍셀은 직접 관측되지 않음
- 하지만 후처리에서 주변 텍셀 값을 복사하여 채움
- 따라서 최종 텍스처에서는 색상이 존재하는 것처럼 보임

### Conclusion
- count=0은 “미관측”을 의미할 뿐, 최종 텍스처에서 반드시 검정으로 남지 않음
- 현재 파이프라인에서는 nearest-neighbor fill로 인해 컬러가 나타남


## 4. 두 픽셀이 하나의 텍셀을 공유하는 문제

<img src="../../docs/assets/meetings/20260429/two_pixel_one_texel.webp" width="800"/> 

### Observations
- 서로 떨어진 camera pixel이 동일 texel에 매핑됨
- 초기에는 UV overlap/버그 의심

### Result
- 두 픽셀은 실제로 동일한 3D 영역(인접 face)을 관측
- view에서 grazing angle(거의 edge-on)로 보임
- 카메라 공간: ~28 px
- UV 공간: 카메라 픽셀 간 거리: 28.2 px
UV 차이: Δu=0.00179 (0.92 texels), Δv=0.00060 (0.31 texels)
→ UV 압축비: 약 29:1 (29 camera px → 1 texel)

### Cause
- UV 면적 대비 screen projection이 과도하게 큼
- 여러 camera pixel이 동일 texel로 quantization됨

### Conclusion
- UV seam/overlap/버그 아님
- grazing angle + UV 압축으로 인한 sampling sparsity 문제

### Solution
- UV 면적 재분배
- multi-view coverage 보완

# Feedback

## 1. 핵심 문제 요약

현재 파이프라인은  
**splatting 기반 UV 업데이트 + post-processing(nearest fill, dilation)** 구조이며,  
이로 인해 다음 문제가 발생:

- 실제 오류를 후처리로 덮음
- unseen texel이 유지되지 않음
- i2i → inverse rendering loop의 신호가 끊김
- 문제 원인에 대한 해석이 왜곡됨

---

## 2. 잘못된 가정 (기존 분석의 오류)

### (1) count=0 해석 오류

- 기존 주장:
  - "count=0은 미관측이며, nearest fill로 색이 채워진다"

- 문제:
  - 해당 영역은 실제로 **관측 가능한 영역**
  - "미관측"이 아니라 **baking 로직 오류 또는 sampling 방식 문제**

→ 결론:  
**count=0을 데이터 문제로 해석한 것이 아니라, 파이프라인 오류를 가린 것**


### (2) "두 픽셀이 하나의 texel 공유" 해석 오류

- 기존 주장:
  - grazing angle + UV 압축 → 동일 texel 매핑

- 문제:
  - 서로 다른 anatomical region (치아 vs 잇몸)
  - 서로 다른 face임이 명확

→ 가능한 원인:
- UV overlap (unwrap bug)
- depth / ray hit 오류
- mapping pipeline 오류

→ 결론:  
**sampling sparsity 문제가 아니라 geometry/UV 매핑 오류 가능성이 더 높음**


### (3) Splatting 기반 접근 자체의 문제

현재 구조: camera pixel → UV → texel (scatter / splatting)

문제:
- UV space는 screen space와 metric이 다름
- 정규화된 kernel 없이 splat → aliasing / hole 발생
- quantization + 충돌 → texel overwrite / skip

→ 결론:
**texture baking에서 splatting은 부적절한 접근**

---

## 3. 핵심 피드백 내용

### (1) Splatting → Gathering 전환

표준 방식: UV texel → screen에서 color sampling (gathering)

- Blender, Substance, nvdiffrast 모두 동일 방식
- texel 기준 sampling → hole 없음

---

### (2) Post-processing 제거

제거 대상:
- distance_transform_edt
- binary_dilation
- nearest neighbor fill

이유:
- unseen texel까지 덮어버림
- i2i가 채워야 할 영역이 사라짐
- 문제를 가리는 band-aid

---

### (3) Seam 문제는 unwrap에서 해결

- per-iteration dilation 금지
- unwrap 단계에서 padding 사용

권장:
- xatlas padding ≈ 8
- Blender margin ≈ 8

---

### (4) 현재 문제의 본질

- UV unwrap에는 **치명적 버그 존재**
- splatting + post-process로 증상을 덮고 있었음
- 분석 과정에서 원인 해석이 왜곡됨

---
